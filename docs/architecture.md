# Техническая архитектура

## Философия

- **Modular First:** на старте монолит, но с чёткими границами доменов. Каждый домен можно будет выделить в микросервис.
- **API First:** всё через REST API. Это позволит позже добавить мобильное приложение, виджеты, внешние интеграции.
- **Build vs. Buy:** не писать с нуля то, что уже решено (чат, стриминг, аутентификация). Собственная разработка — только core value (астро-движок, эзотерический контент).

## Доменная модель

Приложение разбито на домены. Каждый отвечает за свою область и может быть разработан независимо.

```
┌─────────────────────────────────────────────────────────┐
│                      ESTREVIA                           │
├──────────┬──────────┬──────────┬──────────┬────────────┤
│  Astro   │ Cosmos   │  Data    │ Esoteric │    AI      │
│  Engine  │   3D     │  Feed    │   CMS    │   Layer    │
│          │          │          │          │            │
│ Расчёты  │ 3D визу- │ NASA,    │ 777,     │ Claude    │
│ карт,    │ ализация │ USGS,    │ Equinox, │ API,      │
│ эфемери- │ планеты, │ NOAA     │ Thoth,   │ Stability │
│ ды       │ звёзды   │ feeds    │ ToL      │ AI        │
├──────────┴──────────┴──────────┴──────────┴────────────┤
│                   Social Layer                          │
│  Профили, лента, медиа, кастомизация, сообщения        │
├─────────────────────────────────────────────────────────┤
│                   Core Platform                         │
│  Auth, DB, Cache, Storage, API Gateway                  │
└─────────────────────────────────────────────────────────┘
```

### Домены и их сложность

| Домен | Ответственность | Сложность | Фаза |
|-------|----------------|-----------|------|
| **Astro Engine** | Расчёты карт на сервере (Swiss Ephemeris Node.js), sidereal/tropical | Средняя | MVP |
| **Data Feed** | NASA, USGS, NOAA API интеграция | Низкая-средняя | Фаза 2 |
| **Esoteric CMS** | 777, Equinox, Thoth, Tree of Life контент | Низкая (контент) | MVP |
| **Cosmos 3D** | 3D-визуализация, планеты, созвездия | Высокая | Фаза 2 |
| **Social** | Профили, лента, медиа, кастомизация | Очень высокая | Фаза 2 |
| **AI Layer** | Генерация картинок, профили, перевод | Средняя | Фаза 2 |
| **Comms** | Чат, звонки, группы | Очень высокая | Фаза 3 |
| **Music** | Уплоад, стриминг, плейлисты | Очень высокая | Фаза 3 |

## Архитектура MVP

На старте — **монолит на Next.js** с чёткими границами модулей внутри `src/`:

```
src/
├── modules/
│   ├── astro-engine/     # Расчёты карт, Swiss Ephemeris (серверный)
│   │   ├── lib/          # Бизнес-логика расчётов
│   │   ├── components/   # UI: колесо, карта, планетарные часы
│   │   └── api/          # API routes для расчётов
│   │
│   ├── data-feed/        # NASA, USGS, NOAA интеграции (Фаза 2)
│   │   ├── lib/          # Fetch + transform + cache
│   │   ├── components/   # UI: лента событий
│   │   └── api/          # API routes / cron jobs
│   │
│   ├── esoteric/         # Контент: 777, эссе, Tree of Life
│   │   ├── content/      # MDX/JSON файлы с текстами
│   │   ├── components/   # UI: карточки эссе, навигация
│   │   └── lib/          # Утилиты для контента
│   │
│   └── auth/             # Аутентификация (Clerk)
│       ├── components/   # UI: формы, кнопки
│       └── lib/          # Хелперы авторизации
│
├── shared/               # Общие компоненты, утилиты, типы
│   ├── components/       # Layout, Navigation, Theme
│   ├── hooks/            # useTheme, useLocale, useMediaQuery
│   ├── lib/              # Утилиты, constants
│   └── types/            # TypeScript типы
│
└── app/                  # Next.js App Router
    ├── (marketing)/      # Landing, about, pricing
    ├── (app)/            # Основное приложение (за auth)
    │   ├── chart/        # Натальная карта
    │   ├── moon/         # Лунный календарь
    │   ├── hours/        # Планетарные часы
    │   ├── essays/       # Эзотерические эссе
    │   └── feed/         # Лента данных NASA/USGS (Фаза 2)
    └── api/              # API routes
```

### Принципы организации кода

1. **Модуль = домен.** Каждый модуль в `src/modules/` — автономный. Свои компоненты, логика, API.
2. **Зависимости только вниз.** Модули зависят от `shared/`, но не друг от друга. Если нужна связь — через API или shared types.
3. **Next.js App Router как клей.** `app/` директория только роутит и компонует модули. Бизнес-логика — в модулях.
4. **Готовность к разделению.** Любой модуль можно будет вынести в отдельный сервис, заменив прямые вызовы на API вызовы.

## Принципы архитектурных решений

### Build vs. Buy

| Компонент | Решение | Почему |
|-----------|---------|--------|
| Астро-расчёты | Build (Swiss Ephemeris, **серверный**) | Core value, должно быть идеальным. Node.js native bindings (`sweph`) — проверенный, стабильный подход. Расчёт через `POST /api/chart/calculate`, не на клиенте |
| Эзотерический контент | Build (собственные тексты) | Core value, юрид. безопасность |
| Аутентификация | Buy (Clerk) | Решённая проблема, OAuth за 1 день |
| Real-time данные | Build (API integration) | Простые REST вызовы |
| Чат/сообщения | Buy (Stream/Matrix) | Своя реализация = 6+ месяцев |
| 3D визуализация | Build (Three.js) | Кастомный UX, NASA текстуры |
| Музыкальный стриминг | Embed (Spotify/YouTube API) | Лицензии = миллионы |
| AI генерация | Buy (Claude API + Stability) | Не core, быстрый результат |

### Верификация астрологических расчётов

> **Точность — критический фактор доверия. Одна ошибка в градусах = потеря целевой аудитории.**

| Метод | Описание | Допуск |
|-------|----------|--------|
| Эталонное сравнение | Сверка с Astro.com, Solar Fire, Jagannatha Hora | ±0.01° |
| Авто-тесты | 100+ эталонных карт в CI/CD | При каждом деплое |
| Ayanamsa валидация | Lahiri (MVP). Fagan-Bradley, Krishnamurti — Фаза 2 | ±0.01° |
| LLM-верификация | Claude сверяет расчёты с эталонными данными из Astro.com | Автоматически |
| Community фидбэк | Публикация в Reddit/Telegram сидерических сообществах | До публичного запуска |
| Прозрачность | UI показывает «Swiss Ephemeris v2.10 \| Ayanamsa: Lahiri» | — |

## Масштабирование (будущее)

Когда монолит перестанет справляться:

1. **Astro Engine → отдельный сервис.** Вычислительно-интенсивные расчёты на выделенных ресурсах.
2. **Data Feed → фоновые cron jobs.** Уже независим, легко выносится.
3. **Social → отдельный backend.** Самый сложный переход, но модульность поможет.
4. **Comms → сторонний сервис.** Stream/Matrix и так внешние.

На этапе MVP масштабирование не нужно. Vercel + PostgreSQL справятся с первыми 50,000 пользователей.
