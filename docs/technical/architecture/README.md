### Ключевая идея

**Расчёт карты = на устройстве пользователя.** Swiss Ephemeris компилируется в WebAssembly и работает прямо в браузере. Сервер **не участвует** в расчётах. Это даёт:

- **Мгновенный результат** (< 50ms, нет сетевой задержки)
- **Офлайн работу** (карта считается без интернета)
- **Нет нагрузки на сервер** (хоть миллион расчётов одновременно — серверу всё равно)
- **Приватность** (данные рождения не уходят на сервер до сохранения)

Сервер нужен только для: сохранения карт, аутентификации, NASA/USGS данных, подписок, аналитики.

### Правила связей

```
✅ Модуль → Shared          (все используют общие типы)
✅ App Router → Модуль       (страницы собирают модули)
❌ Модуль → Модуль           (нет прямых зависимостей!)
❌ Shared → Модуль           (shared ничего не знает о модулях)
```

**Почему нет связей между модулями:** если astro-engine зависит от data-feed — их нельзя будет разделить потом. Связь идёт через типы в `shared/` и через UI на уровне страницы.

## Что обрабатывает клиент vs сервер

| Действие | Где происходит | Почему |
|----------|---------------|--------|
| Расчёт натальной карты | **Клиент** (WASM) | Скорость, офлайн, приватность |
| Расчёт лунной позиции | **Клиент** (WASM) | Обновляется каждую минуту, нет смысла дёргать сервер |
| Расчёт планетарных часов | **Клиент** (WASM) | Real-time, офлайн |
| Переключение sidereal/tropical | **Клиент** (WASM) | Мгновенный пересчёт, 0ms задержка |
| Отображение эссе | **Клиент** (кэш/MDX) | Статичный контент, кэшируется Service Worker |
| Сохранение карты | **Сервер** (API) | Нужна БД + auth |
| Регистрация / логин | **Сервер** (Clerk) | Auth = серверная логика |
| NASA/USGS данные | **Сервер** (Cron → Redis) | Один запрос к NASA на всех юзеров, не один на каждого |
| Оплата подписки | **Сервер** (Stripe) | PCI compliance, webhooks |
| Аналитика | **Клиент** (PostHog JS) | Events отправляются из браузера |
| Share card (PNG) | **Сервер** (OG Image) | next/og генерирует картинку на сервере |
