# Расчёт натальной карты: от ввода до экрана

> Самый важный техпроцесс в Estrevia. Ошибка здесь = смерть продукта.

---

## Простыми словами

Натальная карта — это «фотография неба» в момент рождения. Мы берём дату, время и место → вычисляем, где стояла каждая планета → показываем на круговой диаграмме (колесо).

Сидерическая карта отличается от тропической на ~24° (прецессия равноденствий). Это значит, что у многих людей знаки «сдвигаются» на один назад.

## Шаг за шагом

### Шаг 1: Ввод данных

```
Пользователь вводит:
  ├── Дата: 15 марта 1990
  ├── Время: 14:30 (опционально)
  └── Город: Moscow, Russia

Что нужно получить:
  ├── Географические координаты: lat 55.7558, lon 37.6173
  ├── Timezone на момент рождения: Europe/Moscow (UTC+3 в 1990)
  └── Julian Day Number: 2447965.97917
```

**Geocoding города:** серверный API `GET /api/v1/cities?q=moscow&limit=10` — поиск по базе ~50K городов (lat/lon/timezone) на сервере. Автокомплит с debounce при вводе. Не Google Maps API — никаких внешних запросов, собственная база.

**Timezone:** библиотека `@vvo/tzdb` (полная IANA tz database) — по координатам определяет IANA timezone ID (например `Europe/Moscow`). Затем `date-fns-tz` конвертирует дату с учётом исторического UTC offset.

> **Важно:** `timezone-lookup` не обрабатывает исторические смены UTC offset (Россия меняла offset в 2011, 2014; Индия, Самоа и др.). `@vvo/tzdb` + `date-fns-tz` корректно работают с IANA tz database, которая содержит все исторические переходы DST и UTC offset для каждого региона.

**Julian Day:** астрономический формат даты — один непрерывный номер дня с 1 января 4713 BC. Swiss Ephemeris работает с Julian Day.

```
JD = date_to_jd(1990, 3, 15, 14, 30, 0, timezone_offset)
   = 2447965.97917
```

### Шаг 2: Swiss Ephemeris (серверный API)

```
Клиент отправляет POST /api/chart/calculate:
  { date, time?, city, lat, lon, timezone }

Сервер (Vercel Function) вызывает sweph (Node.js native bindings):

Входные данные:
  ├── julian_day: 2447965.97917
  ├── planet_id: SE_SUN (0), SE_MOON (1), ... SE_PLUTO (9), ...
  └── flags: SEFLG_SWIEPH | SEFLG_SPEED

Для каждой из 12 небесных тел вызываем:
  swe_calc_ut(julian_day, planet_id, flags)

Результат для каждого тела:
  [0] ecliptic longitude  = 354.254°  (положение на эклиптике)
  [1] ecliptic latitude   = -0.0001°  (отклонение от эклиптики)
  [2] distance            = 0.9935 AU (расстояние от Земли)
  [3] speed in longitude  = 1.0194°/day (скорость движения)
```

**12 тел:** Солнце, Луна, Меркурий, Венера, Марс, Юпитер, Сатурн, Уран, Нептун, Плутон, Северный Лунный Узел, Хирон.

**Время вычисления:** < 50ms для всех 12 тел на сервере. Один вызов swe_calc_ut ~ 2-4ms. Vercel Fluid Compute переиспользует инстансы — cold start только на первый запрос.

### Шаг 3: Тропическая → Сидерическая конверсия

```
Тропическая позиция (из Swiss Ephemeris):
  Солнце = 354.254° (24°15' Рыб тропически)

Ayanamsa (сдвиг) для 1990 года:
  Lahiri:          23.732°
  Fagan-Bradley:   24.736°
  Krishnamurti:    23.828°

Сидерическая позиция (Lahiri):
  354.254° - 23.732° = 330.522° (0°31' Рыб сидерически)

Или напрямую через Swiss Ephemeris:
  swe_set_sid_mode(SE_SIDM_LAHIRI)
  → автоматический пересчёт
```

**Toggle tropical/sidereal:** пересчёт = вычитание одного числа (ayanamsa). Мгновенно, < 1ms. Не нужен повторный вызов Swiss Ephemeris.

### Шаг 4: Знак, градус, минута

```
Абсолютные градусы → знак + позиция в знаке:

  longitude = 330.522°
  sign_index = floor(330.522 / 30) = 11  → Pisces (♓)
  degree_in_sign = 330.522 mod 30 = 0.522°
  
  0.522° = 0° 31' 19.2"
           ↑    ↑    ↑
        градус минуты секунды
  
  Итог: Солнце в 0°31' Рыб (сидерически)
         vs  24°15' Рыб (тропически)
```

### Шаг 5: Дома (если время известно)

```
Если время рождения указано:
  swe_houses_ex(julian_day, lat, lon, 'P')
  // 'P' = Placidus system
  
  → 12 cusps (начала домов):
    House 1 (ASC): 182.5° → 2°30' Весы
    House 2: 210.3° → 0°18' Скорпион
    House 7 (DSC): 2.5° → 2°30' Овен
    House 10 (MC): 92.1° → 2°06' Рак
    ...

Если время НЕ указано:
  → Дома = null
  → ASC = null
  → В UI: "Укажите время рождения для расчёта домов и восходящего знака"

⚠️ POLAR LIMITATION (> 66.5° широты):
  Placidus НЕ работает для полярных широт (Мурманск, Тромсё,
  Рейкьявик, Исландия, вся Антарктида). Куспиды становятся
  undefined, т.к. некоторые градусы эклиптики никогда не восходят.
  
  Fallback:
  → Автоматическое переключение на Whole Sign houses
  → UI-предупреждение: "Для полярных широт используется
     Whole Sign house system вместо Placidus"
  → ASC/MC рассчитываются корректно в любой системе
```

### Шаг 6: Аспекты

```
Для каждой пары планет (66 пар из 12):
  diff = |planet1.longitude - planet2.longitude|
  if diff > 180: diff = 360 - diff

Проверяем каждый тип аспекта:
  ┌─────────────┬───────────┬────────────────┐
  │ Аспект      │ Угол      │ Допуск (орбис) │
  ├─────────────┼───────────┼────────────────┤
  │ Conjunction  │ 0°        │ ±8°            │
  │ Opposition   │ 180°      │ ±8°            │
  │ Trine        │ 120°      │ ±8°            │
  │ Square       │ 90°       │ ±7°            │
  │ Sextile      │ 60°       │ ±6°            │
  │ Quincunx     │ 150°      │ ±3°            │
  │ Semi-sextile │ 30°       │ ±2°            │
  └─────────────┴───────────┴────────────────┘

Пример:
  Солнце = 330.5°, Марс = 242.3°
  diff = |330.5 - 242.3| = 88.2°
  Ближайший аспект: Square (90°)
  Орбис: |90 - 88.2| = 1.8° (< 7° допуска)
  → Square с орбисом 1.8° ✓

Applying vs Separating:
  Если быстрая планета приближается к точному аспекту → applying
  Если удаляется → separating
  (определяется по speed обеих планет)
```

### Шаг 7: Рендеринг SVG колеса

```
Из данных → визуализация:

                    ♈ 0°
                   /    \
              ♓ /        \ ♉
              /            \
         ♒ /    Планеты     \ ♊
         |     на своих      |
    ♑ ── |    градусах в     | ── ♋
         |     кольце        |
         ♐ \                / ♌
              \            /
              ♏ \        / ♍
                   \    /
                    ♎ 180°

SVG элементы:
  1. Внешнее кольцо: 12 секторов по 30° (знаки)
  2. Среднее кольцо: 12 секторов (дома, если известны)
  3. Планеты: точки на соответствующих градусах
  4. Линии аспектов: соединяют пары планет
     - Красные: квадрат, оппозиция (напряжённые)
     - Синие: трин, секстиль (гармоничные)
     - Зелёные: конъюнкция
  5. Глифы: ☉☽☿♀♂♃♄♅♆♇ на позициях планет
```

## Деплой Swiss Ephemeris на Vercel

### Ephemeris файлы (.se1)

Swiss Ephemeris требует файлы эфемерид (`.se1`) для высокоточных расчётов. Без них `sweph` использует встроенные аналитические эфемериды Moshier (точность ~0.1 arcsec ≈ 0.00003° — **достаточна для ±0.01°**).

**Решение для MVP:** Используем **Moshier эфемериды** (встроены в `sweph`, не требуют файлов). Точность 0.00003° значительно превышает наш допуск ±0.01°. Файлы `.se1` не нужны.

**Фаза 2 (опционально):** Если понадобятся DE431 ephemeris для расширенного диапазона дат:
- Файлы `.se1` (~30MB) бандлятся в deployment через `outputFileTracingIncludes` в `next.config.ts`
- Путь указывается через `swe_set_ephe_path()` при инициализации
- Файлы должны быть в `/tmp` или в бандле — Vercel Functions имеет read-only файловую систему

### sweph: нативный C addon на Vercel

`sweph` — нативный C addon (N-API), компилируется через `node-gyp`.

**Что нужно для Vercel:**
- Vercel Functions по умолчанию используют `linux-x64`. Нативные модули должны быть скомпилированы под эту платформу.
- `sweph` предоставляет **prebuilt binaries** через `prebuild-install` — отдельная сборка не нужна.
- В `package.json` зафиксировать точную версию: `"sweph": "x.x.x"` (не `^`).
- **CI:** GitHub Actions `ubuntu-latest` = та же `linux-x64` → тесты на CI валидируют совместимость.
- **Альтернатива:** пакет `swisseph` (JS bindings, тоже native) — у него также есть prebuilt binaries.

> **Верифицировать при первом деплое:** `sweph` загружается и `swe_calc_ut()` возвращает корректные данные в Vercel Function. Тест: `POST /api/chart/calculate` с эталонной датой.

### Geocoding: база городов

Серверный API `GET /api/v1/cities?q=...&limit=10` ищет по базе ~50K городов (крупнейшие по населению, все столицы, региональные центры). Источник: GeoNames (CC BY 4.0, cities15000 — все города с населением > 15K). База хранится на сервере, **не на клиенте** — экономит ~2MB в bundle.

**Ограничение:** Маленькие населённые пункты не найдутся в автокомплите.
**Митигация:** Поле «ручной ввод координат» для опытных пользователей (lat/lon).
**Фаза 2:** Расширить до cities5000 (~140K городов) или подключить внешний geocoding API.

## Performance Budget

| Операция | Цель | Реальность |
|----------|------|-----------|
| Geocode города (клиент) | < 100ms | ~10ms (JSON lookup) |
| API запрос (клиент → сервер) | < 200ms | ~50-150ms (зависит от региона) |
| Timezone lookup (сервер) | < 50ms | ~5ms |
| Swiss Eph: 12 тел (сервер) | < 100ms | ~30-50ms |
| Ayanamsa пересчёт (сервер) | < 1ms | < 0.1ms |
| Аспекты (66 пар) (сервер) | < 10ms | ~2ms |
| SVG рендеринг (клиент) | < 200ms | ~100ms |
| **Итого: ввод → карта** | **< 500ms** | **~250-350ms** |
