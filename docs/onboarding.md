# Онбординг и пользовательские флоу

---

## Принцип: Value Before Registration

> Пользователь получает ценность (расчёт карты) **до** регистрации. Регистрация = сохранение, а не доступ.

```
Визит (Landing / Direct)
  → Ввод даты рождения (без регистрации!)
    → Мгновенный расчёт натальной карты
      → «Сохранить карту?» → Регистрация
        → Сохранённая карта → Эссе → Лунный календарь
          → Premium предложение (через 3-5 сессий)
```

## Флоу 1: Новый посетитель (Landing Page)

```
1. Landing page
   ├── Hero: «Узнай свою настоящую карту звёзд»
   ├── CTA: Ввод даты рождения (3 поля: дата, время, город)
   └── Социальное доказательство (counter: «12,345 карт рассчитано»)

2. Расчёт (серверный API)
   ├── Типичное время: ~300ms (warm function)
   ├── Worst case: ~2-3s (Neon wake-up + Vercel cold start) — см. ниже
   ├── Анимация: тропический знак зачёркивается → сидерический появляется
   ├── WOW-момент: «Ваш настоящий знак — ♓ Рыбы, а не ♈ Овен»
   ├── Результат: натальная карта (sidereal по умолчанию)
   └── Toggle: «Показать тропическую для сравнения»

3. Viral Share (ключевой момент — до регистрации!)
   ├── «Поделиться» → share card «Мой настоящий знак» (без регистрации)
   │   ├── Web Share API (мобильные) → WhatsApp, Telegram, Instagram Stories
   │   ├── «Скопировать ссылку» (десктоп)
   │   └── Deep link: estrevia.app/s/[id] → друг видит результат + CTA
   ├── «Узнать больше про Солнце в Рыбах» → эссе (бесплатное)
   ├── «Сохранить карту» → регистрация (Clerk)
   └── «Сравнить с другом» → (Фаза 2, требует регистрации)

4. Регистрация (Clerk)
   ├── Google OAuth (1 клик)
   ├── Email + magic link
   └── Email + пароль
```

### Cold start: маскировка задержки

Первый запрос после простоя (Neon wake-up ~1s + Drizzle cold start ~50ms + sweph ~50ms) может занять **1-2 секунды**. Это убивает WOW-момент.

**Митигация:**
1. **Анимация расчёта** (~2-3s): пока сервер считает, показываем «skeleton» колеса с вращающимися глифами планет + текст «Рассчитываем позиции 12 небесных тел...». Пользователь воспринимает анимацию как часть опыта, не как задержку.
2. **Warm-up ping**: при загрузке Landing page фоновый `fetch('/api/health')` будит Neon + Vercel Function. К моменту ввода данных (10-30 сек.) функция уже тёплая.
3. **Vercel Fluid Compute**: переиспользует инстансы — cold start только для самого первого запроса после длительного простоя.
4. **Фаза 2**: если cold starts станут проблемой → Neon `always-on` ($19/мес).

### Критические решения

| Решение | Выбор | Почему |
|---------|-------|--------|
| Расчёт без регистрации | ✅ Да | Снижает friction. Конверсия visit→chart 40-60% |
| Sidereal по умолчанию | ✅ Да | Наш differentiator. Toggle для тропической |
| Время рождения обязательно | ❌ Нет | Опционально. Без времени = нет домов, но знаки есть |
| Показать «ваш знак другой» | ✅ Да | WOW-момент. Удержание внимания |
| Share до регистрации | ✅ Да | Viral loop важнее конверсии в регистрацию. Share = бесплатная реклама |

## Флоу 2.5: Viral Share — приход по ссылке друга

> **Это основной канал привлечения.** Пользователь B открывает ссылку от пользователя A.

```
1. Открытие estrevia.app/s/abc123
   ├── OG preview уже показал результат в мессенджере
   └── Страница загружается

2. Страница share
   ├── Карточка результата пользователя A:
   │   «Kirill's real sign is ♓ Pisces (not ♈ Aries)»
   ├── CTA: «What's YOUR real sign?»
   └── Форма ввода даты прямо на странице (без перехода)

3. Расчёт (на той же странице)
   ├── Анимация: тропический → сидерический
   ├── Результат пользователя B
   ├── «Поделиться» → создаёт свою карточку /s/[newId] → LOOP
   ├── «Узнать больше» → эссе
   └── «Сохранить карту» → регистрация
```

**Ключевой принцип:** минимум шагов от ссылки до собственного результата. Никакой регистрации, никаких попапов. Дата → результат → share. Три тапа.

## Ошибки и Edge Cases

| Сценарий | Как обрабатываем |
|----------|-----------------|
| Время рождения неизвестно | Показываем карту без домов. Пояснение: «Время рождения влияет на дома и восходящий знак» |
| Город не найден | Автокомплит + ручной ввод координат |
| Дата в будущем | Блокируем. «Натальная карта рассчитывается для прошлых дат» |
| Дата до 1800 | Предупреждение: «Точность эфемерид снижается для дат до 1800 года» |
| Offline | MVP не поддерживает offline-режим. Показываем стандартное браузерное поведение при потере сети |
