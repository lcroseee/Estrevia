# Фреймворк: Next.js

## Что это простыми словами

Next.js — это «оболочка» вокруг React, которая добавляет то, чего в голом React нет: серверный рендеринг (страницы готовы до загрузки JS), маршрутизацию (файл = страница), API routes (бэкенд прямо в проекте), и оптимизацию (картинки, шрифты, кэширование). Это как получить каркас дома вместо кучи кирпичей.

---

## Почему Next.js

| Причина | Для Estrevia |
|---------|-------------|
| **SSR/ISR** | Эссе по позициям рендерятся на сервере → мгновенная загрузка + SEO |
| **API Routes** | Бэкенд (расчёты, NASA API, auth) прямо в проекте, без отдельного сервера |
| **App Router** | Layouts, loading states, error boundaries — для сложного UI астрологического колеса |
| **PWA-ready** | Service Worker + manifest → offline расчёт карт |
| **Vercel** | Zero-config deploy, preview для каждого PR |
| **Экосистема** | 138K stars на GitHub, 44M downloads/нед. Любой вопрос уже отвечен на StackOverflow |

## Альтернативы: почему не они

### Remix (React Router v7)

| Аспект | Подробности |
|--------|------------|
| **Что это** | Был конкурент Next.js от создателей React Router. Теперь слился с React Router v7 |
| **Почему не подходит** | Бренд заброшен, сообщество в растерянности. На Remix Jam 2025 показали прототип v3 **без React** (!). Рискованная ставка |
| **Что пишут** | «Please reconsider abandoning the Remix name» — GitHub discussion с сотнями комментариев. Разочарование |
| **Когда был бы лучше** | Если бы нужна мутация-first архитектура (формы, CRUD). Для Estrevia — не приоритет |

### Astro

| Аспект | Подробности |
|--------|------------|
| **Что это** | Фреймворк для контент-сайтов. Отправляет ноль JS по умолчанию, «островная» архитектура |
| **Почему не подходит** | Estrevia — **интерактивное приложение** (колесо карты, real-time часы, toggle). Astro оптимален для блогов и документации, не для app-like UI |
| **Плюсы** | Куплен Cloudflare (янв. 2026). Lighthouse 95-100 из коробки. 2-3x быстрее Next.js для контент-страниц |
| **Когда был бы лучше** | Если бы Estrevia был чисто контентным сайтом (только эссе + лендинг). Но у нас расчёты, interactive charts, auth |

### SvelteKit

| Аспект | Подробности |
|--------|------------|
| **Что это** | Фреймворк на Svelte — компилятор вместо runtime. Svelte 5 добавил «runes» (новая система реактивности) |
| **Почему не подходит** | Экосистема в 20x меньше React. Найти Svelte-разработчика в 5x сложнее. Для соло-проекта — рискованно |
| **Плюсы** | Бандлы на 15-30% меньше, DX лучше чем React. Разработчики в восторге |
| **Когда был бы лучше** | Если бы у нас была команда из Svelte-энтузиастов. Или если бы не нужна была совместимость с React-экосистемой (shadcn/ui, Three.js, Clerk) |

## Известные проблемы Next.js

### 1. Memory leaks в Docker (GitHub #88603)

В контейнерах память растёт линейно до OOM-kill. Актуально для Kubernetes/Docker.

**Влияние на нас:** минимальное. Мы на Vercel (managed), не в Docker.

### 2. Безопасность (CVE-2025-55182, CVSS 10.0)

В конце 2025 — 6 CVE, включая критическую уязвимость App Router RSC. Все пропатчены.

**Влияние на нас:** всегда обновляться до последней стабильной версии. Vercel обновляет автоматически.

### 3. App Router сложнее Pages Router

Ментальная модель Server Components + Client Components + Server Actions — непростая для новичков.

**Влияние на нас:** кривая обучения. Но App Router — будущее Next.js, Pages Router deprecated.

### 4. Vercel lock-in

ISR, `next/image`, edge middleware, Vercel-specific storage — всё привязывает к Vercel. Self-hosting работает через `output: 'standalone'`, но теряешь ISR кэширование и image optimization.

**Влияние на нас:** осознанный trade-off. На MVP lock-in приемлем ради скорости. При росте — можно мигрировать на Cloudflare или self-hosted.

### 5. Build times

Большие проекты (500+ страниц) могут собираться 5-10 минут. Turbopack ускоряет dev, но prod build всё ещё webpack.

**Влияние на нас:** на MVP (~150 страниц) не проблема. Следить при росте контента.

## Вердикт

**Next.js — правильный выбор для Estrevia.** Не потому что лучший фреймворк в мире, а потому что:

1. React экосистема = shadcn/ui, Three.js, Clerk, PostHog — всё работает из коробки
2. Один разработчик может делать frontend + backend + API в одном проекте
3. Vercel deployment = zero-config
4. Наибольшее сообщество = быстрое решение любых проблем
5. SEO для 120+ эссе = SSR/ISR критически важен

**Риск:** Vercel lock-in. **Митигация:** держать бизнес-логику в `src/modules/`, не в Next.js-specific коде. При необходимости — миграция модулей, не всего приложения.
