# Разрешение часовых поясов: от локального времени к UTC

> Критический процесс. Ошибка в timezone = неправильная карта = потеря доверия пользователя.

---

## Почему это важно

Натальная карта рассчитывается для конкретного момента времени в UTC. Пользователь вводит **локальное** время рождения — например, 23:42 в Leamington Spa, UK, 12 октября 1875 года. Нужно точно определить: действовал ли BST (UTC+1) или GMT (UTC+0) в этот момент.

Ошибка на 1 час сдвигает Луну на ~0.5°, меняет куспиды домов и может изменить Асцендент на другой знак. Для пользователя это означает: «приложение врёт».

---

## Проблема: исторические часовые пояса

Часовые пояса — не константа. Они менялись, и менялись хаотично:

| Регион | Проблема |
|--------|----------|
| **США до 1883** | Стандартные часовые пояса не существовали. Каждый город использовал local solar time |
| **Мир до 1884** | Международная конвенция о часовых поясах ещё не принята |
| **Индия до 1947** | Несколько offset: Bombay Time (UTC+4:51), Calcutta Time (UTC+5:53:28) |
| **Россия** | Менялась неоднократно — отмена DST (2011), возврат к зимнему времени (2014) |
| **Крым (2014)** | Переход с `Europe/Kiev` (UTC+2) на `Europe/Moscow` (UTC+3) |
| **Самоа (2011)** | Перескочила через International Date Line — пропущен целый день (30 декабря) |

IANA Time Zone Database (`tzdata`) содержит все исторические переходы. Это единственный надёжный источник.

## Архитектура решения

### Общий поток

```
Пользователь вводит:                  Что происходит:
─────────────────────                  ────────────────

1. Город: "Leamington Spa"     →  Автокомплит → lat/lon + IANA zone ID
                                    lat: 52.2919, lon: -1.5367
                                    zone: "Europe/London"

2. Дата: 12 October 1875       →  Вместе с zone ID определяем
   Время: 23:42                    исторический UTC offset на эту дату

3. Europe/London + 1875-10-12  →  GMT (UTC+0) — BST введён только в 1916
   23:42 local                     Offset = +00:00

4. UTC time: 1875-10-12T23:42Z →  Конвертируем в Julian Day
                                    → передаём в Swiss Ephemeris (sweph)
```

### Шаг 1: Город → IANA timezone ID

Серверная база ~50K городов (доступна через `GET /api/v1/cities`) содержит для каждого города:
- `name`, `country`
- `lat`, `lon`
- `timezone` — строка IANA zone ID (например `"Europe/London"`, `"Asia/Kolkata"`)

Пользователь выбирает город из автокомплита (серверный поиск с debounce). Мы получаем IANA zone ID без внешних API-запросов.

### Шаг 2: IANA zone + дата → UTC offset

Библиотека `@vvo/tzdb` предоставляет полную IANA tz database, включая все исторические переходы DST и offset для каждой зоны. `date-fns-tz` использует эти данные для конвертации.

```typescript
import { getTimezoneOffset } from "date-fns-tz";

// Получаем offset в миллисекундах для конкретной даты и зоны
const offsetMs = getTimezoneOffset("Europe/London", new Date(1875, 9, 12));
// → 0 (GMT, BST ещё не введён)

const offsetMs2 = getTimezoneOffset("Europe/London", new Date(2024, 6, 15));
// → 3600000 (BST, UTC+1)
```

### Шаг 3: Локальное время → UTC

```typescript
function resolveTimezone(
  date: string,       // "1875-10-12"
  time: string | null, // "23:42" или null (время неизвестно)
  ianaZone: string     // "Europe/London"
): { utcDate: Date; tzOffset: number } | null {
  // Время неизвестно → пропускаем timezone, дома = null
  if (!time) return null;

  const [year, month, day] = date.split("-").map(Number);
  const [hours, minutes] = time.split(":").map(Number);

  // Собираем локальную дату
  const localDate = new Date(year, month - 1, day, hours, minutes);

  // Получаем исторический offset для этой зоны на эту дату
  const offsetMs = getTimezoneOffset(ianaZone, localDate);
  const offsetHours = offsetMs / 3600000;

  // Конвертируем в UTC
  const utcDate = new Date(localDate.getTime() - offsetMs);

  return { utcDate, tzOffset: offsetHours };
}
```

### Шаг 4: UTC → Julian Day → Swiss Ephemeris

```typescript
import { swe_julday, SE_GREG_CAL } from "sweph";

const { utcDate } = resolveTimezone("1875-10-12", "23:42", "Europe/London");

const jd = swe_julday(
  utcDate.getUTCFullYear(),
  utcDate.getUTCMonth() + 1,
  utcDate.getUTCDate(),
  utcDate.getUTCHours() + utcDate.getUTCMinutes() / 60,
  SE_GREG_CAL
);

// Далее: swe_calc_ut(jd, planetId, flags)
```

## Edge cases

### 1. Рождение во время перехода DST (ambiguous time)

При переводе часов назад (fall-back) локальное время 2:00-3:00 повторяется дважды. Например, в США в первое воскресенье ноября: 1:30 AM EST и 1:30 AM EDT — два разных момента UTC.

**Решение MVP:** показываем предупреждение: «Время рождения попадает в период перехода DST. Карта может отличаться на ~1°. Уточните, было ли это летнее или зимнее время.» Рассчитываем по стандартному (зимнему) времени как default.

**Phase 2:** UI-выбор «летнее / зимнее время» при обнаружении ambiguous time.

### 2. Рождение до стандартных часовых поясов

До 1883 (США) / 1884 (международный стандарт) города использовали local mean time (LMT) — солнечное время по долготе.

**Решение:** IANA tz database содержит LMT offsets для крупных городов до введения стандартных зон. `@vvo/tzdb` корректно возвращает эти значения. Для мелких городов без записи в IANA — используем ближайший крупный город той же зоны.

### 3. Города, сменившие часовой пояс

Крым (2014): `Europe/Simferopol` в IANA database отражает переход UTC+2 → UTC+3. Рождение в Симферополе в 2013 и 2015 корректно получат разные offsets.

Самоа (2011): `Pacific/Apia` в IANA database отражает пропуск 30 декабря 2011. Рождение в эту дату невозможно — показываем ошибку валидации.

### 4. Время рождения неизвестно

Если пользователь не указал время:
- `resolveTimezone()` возвращает `null`
- Планеты рассчитываются на полдень UTC (12:00) указанной даты — стандартная практика в астрологии
- Дома, Асцендент, MC = `null`
- В UI показываем упрощённую карту без домов

### 5. Субминутная точность

MVP принимает время с точностью до минуты (`HH:MM`). Секунды не запрашиваем — подавляющее большинство людей не знают секунд рождения. Ошибка ±30 секунд даёт максимум ~0.008° сдвига Луны — пренебрежимо.

## Зависимости

| Пакет | Назначение | Размер |
|-------|-----------|--------|
| `@vvo/tzdb` | Полная IANA tz database с историческими переходами | ~1.5 MB (server-side only) |
| `date-fns-tz` | Конвертация дат с учётом timezone | ~15 KB |

Обе библиотеки используются **только на сервере** (Vercel Function). Клиент отправляет локальное время + IANA zone ID, сервер выполняет конвертацию.
